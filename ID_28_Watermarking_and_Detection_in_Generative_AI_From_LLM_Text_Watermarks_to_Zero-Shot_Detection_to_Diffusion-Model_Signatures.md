# Watermarking and Detection in Generative AI: From LLM Text Watermarks to Zero-Shot Detection to Diffusion-Model Signatures

## What This Report Teaches

This report explains three different strategies for identifying AI-generated content.

1. **A Watermark for Large Language Models** is a **proactive** method: the model owner intentionally changes generation so the output carries a hidden statistical signature that can later be detected. ([ar5iv][1])
2. **DetectGPT** is a **reactive** method: it does not require any watermark to have been added in advance. Instead, it tries to tell whether a passage was generated by a given language model by checking how the model’s probability changes under small rewrites. ([ar5iv][2])
3. **Stable Signature** is a proactive watermarking method for **latent diffusion image models**: it fine-tunes part of the generator so that all generated images contain a hidden recoverable signature. ([ar5iv][3])

By the end, you should understand the difference between **watermarking** and **detection**, how each method works step by step, why they solve different problems, and how to explain their trade-offs in an interview. ([ar5iv][1])

## Key Takeaways

* **Watermarking and detection are not the same thing.** Watermarking adds a hidden signal during generation, while detection tries to infer machine generation afterward. This matters because watermarking requires cooperation from the model provider, but detection can be applied to already-generated content. The practical implication is that proactive and reactive methods solve different governance problems. ([ar5iv][1])

* **The LLM watermark paper changes token sampling, not model training.** It randomly marks some next-token options as “green” and softly biases generation toward them, then detects the resulting statistical excess of green tokens with a z-test. This matters because it makes watermarking possible without retraining the whole LLM. The practical implication is that a provider can add provenance signals at decoding time with low overhead. ([ar5iv][1])

* **DetectGPT is not a watermark at all.** It relies on the observation that model-generated text tends to lie in regions where small meaning-preserving perturbations reduce the source model’s log probability. This matters because it gives a zero-shot detector that does not need a separate classifier or pre-watermarked outputs. The practical implication is that it is useful when you suspect model-generated text but had no chance to watermark it beforehand. ([ar5iv][2])

* **Stable Signature roots the watermark inside the diffusion model’s decoder.** It fine-tunes the latent decoder so generated images automatically carry a hidden binary signature recoverable by a watermark extractor. This matters because it embeds provenance into the model itself rather than only into a post-processing step. The practical implication is that every generated image can become traceable to the originating model or even a distributed model instance. ([ar5iv][3])

* **Each paper assumes a different threat model.** The LLM watermark assumes the generator cooperates; DetectGPT assumes access to the source model’s probabilities or a close surrogate; Stable Signature assumes access to the diffusion model for fine-tuning and a private or protected extractor for robust attribution. This matters because “best method” depends on what access you have. The practical implication is that interviews should compare methods by deployment assumptions, not just headline accuracy. ([ar5iv][1])

* **Robustness is the core challenge in all three papers.** Text watermarking is weakened by editing or paraphrasing, DetectGPT is computationally heavy and best in a white-box setting, and Stable Signature can be attacked if the extractor leaks or if the model is purified. This matters because provenance systems fail if attackers can cheaply erase the signal. The practical implication is that watermarking and detection are security problems as much as ML problems. ([ar5iv][1])

* **These papers span both text and images, but the underlying idea is the same: make synthetic content statistically distinguishable from natural content.** This matters because provenance is a cross-modal problem. The practical implication is that the same interview answer can often be structured around “where is the signal inserted, how is it detected, and what assumptions does it require?” ([ar5iv][1])

## Background and Foundations

Generative AI creates content that can be highly fluent, realistic, and scalable. That creates value, but it also creates a provenance problem: once a piece of text or an image exists, how can you tell whether it came from a human or from a model? The three papers here answer that question in different ways. ([ar5iv][1])

A useful first distinction is between **passive detection** and **active watermarking**.

* **Passive detection** means you examine the content after the fact and look for traces that naturally arise from model generation. DetectGPT is in this category. ([ar5iv][2])
* **Active watermarking** means the generator intentionally embeds a signal during generation so later detection becomes easier. The LLM watermark paper and Stable Signature are in this category. ([ar5iv][1])

A second distinction is between **content-level** and **model-rooted** watermarking.

* In the LLM watermark paper, the watermark is added during **token sampling**. It lives in the generated text’s token distribution. ([ar5iv][1])
* In Stable Signature, the watermark is rooted in the **model parameters**, specifically the latent decoder, so all generated images inherit the signature. ([ar5iv][3])

A third distinction is between **detection of origin** and **identification of source**.

* Some systems only answer: “Was this AI-generated?” ([ar5iv][1])
* Others can support stronger attribution: “Was this generated by my model?” or even “Which distributed copy of my model generated this?” Stable Signature explicitly discusses both detection and identification. ([ar5iv][3])

Historically and conceptually, the three papers fit together well. The LLM watermark paper asks how to make future LLM outputs detectable by design. DetectGPT asks how to detect machine-generated text even when no watermark was added. Stable Signature extends the provenance problem into image generation by embedding a hidden signature inside a latent diffusion model. A reasonable interpretation is that together they span the main provenance design space: **inject a signal**, **infer a signal**, or **root a signal in the model itself**. That framing is a synthesis, but it follows closely from the methods. ([ar5iv][1])

## Core Concepts Explained

### Watermarking

A **watermark** is a hidden signal intentionally inserted into generated content so it can later be recognized. It is not meant to be visible or obvious to humans. In the LLM paper, the watermark is statistical: generated text uses a suspiciously high fraction of special “green” tokens. In Stable Signature, the watermark is a recoverable hidden binary signature inside generated images. This matters because watermarking is proactive: it only works if the generator is designed to cooperate. ([ar5iv][1])

### Detection

A **detector** tries to decide whether content was machine-generated. DetectGPT is a detector without watermarking. The detector in the LLM watermark paper is a statistical test for an intentionally inserted signal. Stable Signature also uses a detector, but only after a watermark extractor decodes a candidate message from the image. This matters because “detector” describes the decision rule, while “watermark” describes where the signal came from. ([ar5iv][1])

### Green list and red list

In the LLM watermark paper, before each token is sampled, the vocabulary is randomly partitioned into a **green list** and a **red list** using a pseudo-random process seeded from recent context. The watermark then biases the model toward green-list tokens. The detector later counts how often green tokens appear and checks whether that count is implausibly high under normal writing. This matters because the watermark is not a visible code word. It is a repeated small statistical preference across many token decisions. ([ar5iv][1])

### z-test

The LLM watermark paper uses a **one-proportion z-test**. In plain English, it asks: if this were ordinary human text, how surprising would it be to see this many green tokens? If the answer is “extremely surprising,” the detector flags the text as watermarked. This matters because it produces interpretable p-values rather than only a black-box score. ([ar5iv][1])

### Probability curvature

DetectGPT’s key concept is **probability curvature**. The paper argues that model-generated text tends to lie near local maxima of the source model’s log-probability function. So when you make small meaning-preserving perturbations, those perturbations usually get lower log probability than the original generated passage. Human-written text does not show the same pattern as strongly. This matters because the detector uses a geometric property of the model’s probability surface rather than a separate learned classifier. ([ar5iv][2])

### Perturbation discrepancy

DetectGPT measures a **perturbation discrepancy**: how much the source model’s log probability drops when the passage is lightly rewritten. The passage is perturbed by masking some spans and filling them with a separate model such as T5. If the original passage is much more probable than its perturbations, DetectGPT treats that as evidence of machine generation. This matters because it turns a vague intuition about “model-like text” into a measurable score. ([ar5iv][2])

### White-box setting

DetectGPT is mainly evaluated in a **white-box** setting, meaning the detector can query log probabilities from the source model. The paper notes that this does not necessarily require model weights, but it does require scoring access, which many APIs may not expose. This matters because DetectGPT is not universally deployable against any arbitrary model endpoint. ([ar5iv][2])

### Latent decoder

Stable Signature works with **latent diffusion models**, where image generation happens via a latent representation and then a decoder turns latent vectors into pixels. The paper fine-tunes the **latent decoder** so that decoded images automatically contain a hidden signature while leaving the diffusion process itself unchanged. This matters because the watermark is rooted in the model’s rendering component rather than only in output post-processing. ([ar5iv][3])

### Watermark extractor

Stable Signature relies on a **watermark extractor**, a learned network that reads an image and predicts the hidden binary message. The method is trained in two phases: first train the extractor, then fine-tune the diffusion model’s decoder so generated images reliably carry a chosen signature that the extractor can recover. This matters because detection depends on both the generator and the extractor working together. ([ar5iv][3])

## Step-by-Step Technical Walkthrough

## 1. A Watermark for Large Language Models

### Stage 1: Start with ordinary next-token generation

**Input:** a prompt and a standard language model ready to sample the next token.
**What happens:** the LLM produces logits over the vocabulary as usual.
**Output:** a normal next-token distribution before watermarking.
**Why this step exists:** the watermark is designed to work at decoding time, not by retraining the whole model.
**Trade-off:** because it works at generation time, it only helps for content produced by cooperating systems. ([ar5iv][1])

### Stage 2: Randomly partition the vocabulary

**Input:** the current token context.
**What happens:** the system hashes recent context and uses it to pseudo-randomly split the vocabulary into a green list and a red list.
**Output:** a context-dependent random partition of tokens.
**Why this step exists:** the watermark should not use a fixed set of favorite words that would be easy to notice or strip out.
**Trade-off:** context dependence strengthens secrecy, but it also means edits can affect later token classifications. ([ar5iv][1])

### Stage 3: Bias generation toward green tokens

**Input:** the original logits and the green/red partition.
**What happens:** instead of banning red tokens outright in the more practical soft version, the algorithm adds a constant bias to green-token logits before sampling.
**Output:** a slightly modified next-token distribution that prefers green tokens.
**Why this step exists:** a soft bias preserves fluency better than hard constraints.
**Trade-off:** stronger bias makes detection easier but can hurt text quality; weaker bias preserves quality but weakens the signal. The paper especially warns that low-entropy sequences are harder to watermark well. ([ar5iv][1])

### Stage 4: Generate the text

**Input:** the biased token distribution at each step.
**What happens:** generation proceeds normally, but the sequence accumulates a statistical excess of green tokens.
**Output:** watermarked text that still reads naturally to humans.
**Why this step exists:** the watermark must be invisible to casual readers while remaining detectable statistically.
**Trade-off:** extremely predictable text is difficult, because when there is only one plausible next token the model has little freedom to bias sampling without harming quality. ([ar5iv][1])

### Stage 5: Detect the watermark later

**Input:** a candidate text passage.
**What happens:** the detector reconstructs the green/red partitions token by token, counts green-token occurrences, and computes a one-proportion z-test.
**Output:** a detection decision plus a p-value-style confidence score.
**Why this step exists:** the goal is not only to mark text but to verify provenance later without rerunning the original LLM.
**Trade-off:** the detector works best on higher-entropy passages and weakens as text is heavily edited. ([ar5iv][1])

### Stage 6: Consider robustness and attacks

The paper argues that removing the watermark from long passages requires modifying a substantial fraction of tokens, roughly on the order of a quarter or more in the analysis given. It also discusses public and private watermarking modes, where private mode keeps the key secret and makes removal harder. This matters because watermarking is only useful if attackers cannot cheaply erase it. ([ar5iv][1])

## 2. DetectGPT

### Stage 1: Take a candidate passage and a source model

**Input:** a passage and the model you suspect may have generated it.
**What happens:** the detector assumes it can score the passage under that model.
**Output:** a baseline log probability for the original passage.
**Why this step exists:** DetectGPT measures how “locally optimal” the passage is under the source model.
**Trade-off:** this is why the method is strongest in the white-box or scoring-access setting. ([ar5iv][2])

### Stage 2: Generate perturbations

**Input:** the passage.
**What happens:** DetectGPT masks small spans and fills them using a separate perturbation model such as T5. In the main setup, it masks randomly sampled two-word spans until 15% of the words are masked and uses many perturbation samples.
**Output:** multiple nearby rewrites of the original passage.
**Why this step exists:** the detector needs nearby alternatives to probe the local shape of the source model’s probability function.
**Trade-off:** better perturbations improve detection, but generating many of them is expensive. ([ar5iv][2])

### Stage 3: Score the perturbations under the source model

**Input:** the original passage and its perturbations.
**What happens:** the source model assigns log probabilities to the original and to each perturbed passage.
**Output:** a set of probability comparisons.
**Why this step exists:** machine-generated text is hypothesized to have an unusually large average log-probability drop under perturbation.
**Trade-off:** this relies on the source model or a close surrogate being available for scoring. ([ar5iv][2])

### Stage 4: Compute the discrepancy score

**Input:** the original and perturbation log probabilities.
**What happens:** DetectGPT computes the average drop, optionally normalized by the perturbation variance.
**Output:** a curvature-inspired detection score.
**Why this step exists:** the score is meant to approximate whether the passage sits in a negative-curvature region of the model’s log-probability surface.
**Trade-off:** it is more principled than simple probability thresholding, but more computationally expensive. ([ar5iv][2])

### Stage 5: Classify the passage

**Input:** the discrepancy score and a threshold.
**What happens:** if the score is high enough, the passage is labeled as probably model-generated.
**Output:** a binary or thresholded decision.
**Why this step exists:** DetectGPT is designed as a zero-shot detector without training a separate classifier.
**Trade-off:** it does not provide the same kind of cryptographic provenance guarantee as watermarking; it is statistical evidence, not proof of intentional marking. ([ar5iv][2])

### Stage 6: Handle revisions and distribution shift

The paper shows DetectGPT remains reasonably strong under partial simulated revisions and often generalizes better than supervised detectors to new domains and languages. But it is still computationally heavy, and performance falls when the source model is unknown or only a surrogate is available. ([ar5iv][2])

## 3. Stable Signature

### Stage 1: Decide on the signature and threat model

**Input:** a model owner who wants future generated images to be attributable.
**What happens:** the owner chooses a binary signature and a detection or identification goal.
**Output:** a target message to embed in future images.
**Why this step exists:** the method is designed both for binary detection (“from my model or not”) and stronger identification (“which distributed model copy generated this?”).
**Trade-off:** stronger attribution schemes require careful control of signatures and false-positive rates. ([ar5iv][3])

### Stage 2: Pre-train a watermark extractor

**Input:** images and binary messages.
**What happens:** the paper first trains a watermark extractor network, using a classical watermarking setup inspired by HiDDeN, so it can recover a message even after common image transformations.
**Output:** an extractor that can decode soft bits from images.
**Why this step exists:** later the diffusion model will be tuned so its outputs become decodable by this extractor.
**Trade-off:** the extractor is security-sensitive; the paper notes that leaking it can help attackers. ([ar5iv][3])

### Stage 3: Fine-tune the latent decoder

**Input:** the pretrained latent diffusion model and the target signature.
**What happens:** only the decoder is fine-tuned so that images decoded from latent vectors carry the chosen signature. The paper emphasizes that it does not change the diffusion process itself.
**Output:** a watermarked generator.
**Why this step exists:** fine-tuning a small, focused part of the model is cheaper and more compatible with existing latent diffusion pipelines.
**Trade-off:** this is model-rooted watermarking, so it requires access to model fine-tuning rather than just post-hoc detection. ([ar5iv][3])

### Stage 4: Generate images as usual

**Input:** prompts or other latent diffusion tasks.
**What happens:** the model runs its normal diffusion generation pipeline, and the decoder produces images that now contain the hidden signature.
**Output:** natural-looking images with an invisible embedded watermark.
**Why this step exists:** the watermark should be native to generated outputs, not an afterthought added outside the generator.
**Trade-off:** the perceptual distortion must remain low enough that users do not notice degradation. ([ar5iv][3])

### Stage 5: Extract the signature and test statistically

**Input:** a suspect image.
**What happens:** the extractor predicts the message bits, and a statistical test checks whether the extracted message matches the expected signature closely enough to reject the null hypothesis.
**Output:** a detection or attribution decision with a controlled false-positive rate.
**Why this step exists:** a raw decoded bit string is noisy; the statistical test turns it into a principled decision rule.
**Trade-off:** the guarantees depend on assumptions about extractor outputs on non-generated images behaving like near-random bits. ([ar5iv][3])

### Stage 6: Consider attacks

The paper studies image-level attacks such as cropping and stronger attacks involving the extractor, as well as network-level attacks such as “model purification,” where an attacker fine-tunes the model to weaken or remove the watermark. This matters because rooting the watermark in the model helps, but does not make it invulnerable. ([ar5iv][3])

## Paper-by-Paper Explanation

## 1. *A Watermark for Large Language Models*

### The problem addressed

The paper asks how a provider of a proprietary LLM can make its generated text detectable later, without retraining the model and without needing a detector that has access to model parameters or the serving API. The broader motivation is harm reduction for fake news, spam, cheating, and synthetic data contamination. ([ar5iv][1])

### The method used

The method partitions the vocabulary into green and red tokens at each generation step using a pseudo-random process based on context, then softly boosts the logits of green tokens before sampling. Detection later reconstructs those same partitions and applies a one-proportion z-test to the observed green-token count. ([ar5iv][1])

### The main innovation

The main innovation is a watermark that is **statistical, decoding-time, and open-detector friendly**. The detector can be public and does not need access to model weights or to a secret internal classifier. That is a big practical difference from approaches that require training a dedicated detector model. ([ar5iv][1])

### The main findings

The paper says the watermark can be detected from short spans, in some cases as few as 25 tokens, with statistically improbable false positives, and can be embedded with negligible text-quality impact. It also analyzes detectability and robustness formally and empirically on OPT-family models. ([ar5iv][1])

### The limitations

The paper is explicit that **low-entropy sequences** are difficult to watermark well. Very predictable continuations leave the model little freedom to bias token choice without harming quality, and they are also intrinsically hard to distinguish from human writing. The paper also treats robustness as statistical rather than cryptographic certainty and discusses the need for substantial editing to remove the signal from long passages. ([ar5iv][1])

### What changed compared with earlier work

Compared with generic AI-text detectors, this method is proactive and provider-controlled. Compared with retraining-based watermarking ideas, it works at sampling time and requires no model retraining. That makes it an important early paper in practical LLM provenance. ([ar5iv][1])

## 2. *DetectGPT: Zero-Shot Machine-Generated Text Detection using Probability Curvature*

### The problem addressed

DetectGPT asks how to detect model-generated text **without** watermarking and **without** training a new supervised detector for each new source model. The central challenge is that supervised detectors often overfit to the domains, topics, and models they were trained on. ([ar5iv][2])

### The method used

Given a candidate passage and a source model, DetectGPT creates many small perturbations with a mask-filling model such as T5, scores both the original and the perturbations under the source model, and measures how much the original stands out as locally higher probability. A large discrepancy suggests machine generation. ([ar5iv][2])

### The main innovation

The main innovation is the use of **local probability curvature** as a detection signal. Instead of learning a detector, the method exploits a geometric property of the source model’s own probability function. That is why it is zero-shot. ([ar5iv][2])

### The main findings

The paper reports that DetectGPT improves fake-news detection AUROC on GPT-NeoX 20B generations from 0.81 for the strongest zero-shot baseline to 0.95. It also reports the best zero-shot detection performance in 14 of 15 model-dataset combinations and shows that it stays above 0.8 AUROC even when nearly a quarter of the text has been replaced in its simulated revision setting. ([arXiv][5])

### The limitations

DetectGPT is most suited to the white-box or scoring-access setting and is computationally expensive because it requires generating and scoring many perturbations per passage. It also depends on having a good perturbation function, and performance drops when the source model is unknown and only a surrogate is available. ([ar5iv][2])

### What changed compared with earlier work

Compared with supervised detectors, DetectGPT avoids training data and tends to generalize better across domains and languages. Compared with watermarking, it is reactive and can be used after the fact, but it cannot offer intentional provenance guarantees because no signal was embedded at generation time. ([ar5iv][2])

## 3. *The Stable Signature: Rooting Watermarks in Latent Diffusion Models*

### The problem addressed

Stable Signature asks how a latent diffusion model can be modified so that **all generated images** carry an invisible watermark that remains detectable even after some common image transformations. The motivation is responsible deployment and provenance for image generation models. ([ar5iv][3])

### The method used

The method first trains a watermark extractor network, then fine-tunes only the latent decoder of the diffusion model so generated images decode to the desired binary signature. Later, a detector extracts the bits and applies a statistical test for detection or identification. ([ar5iv][3])

### The main innovation

The main innovation is **rooting the watermark in the latent decoder** rather than adding it afterward as a separate image-editing step. The paper emphasizes that this requires no architectural change and does not alter the diffusion process itself. ([ar5iv][3])

### The main findings

The paper reports robustness to image modifications and gives a headline example: it can still detect the origin of an image cropped down to 10% of its content with 90%+ accuracy at a false positive rate below `10^-6`. It also supports both simple detection and stronger identification of which distributed model copy produced the image. ([ar5iv][3])

### The limitations

The extractor is security-sensitive, and the paper explicitly notes that if an attacker gains access to it they can perform stronger attacks. It also studies model purification, where the model is fine-tuned to reduce watermark embedding. So the method is strong, but not immune to adaptive adversaries. ([ar5iv][3])

### What changed compared with earlier work

Compared with image-space watermarking added after generation, Stable Signature moves the watermarking mechanism inside the latent diffusion model itself. Compared with text watermarking at token sampling time, it is a model-parameter-rooted watermark for images. ([ar5iv][3])

## Real-World System and Application

A real-world provenance system could combine these ideas rather than choosing only one.

1. A commercial LLM provider could use a **generation-time watermark** like the green-token method for its own outputs. ([ar5iv][1])
2. Platforms receiving suspect text from unknown sources could use a **reactive detector** like DetectGPT when they do not control generation. ([ar5iv][2])
3. An image model distributor could use a **model-rooted watermark** like Stable Signature to trace generated images back to a deployed or distributed model. ([ar5iv][3])

That suggests a practical architecture:

| Deployment layer              | Likely method    | Why                                                                    |
| ----------------------------- | ---------------- | ---------------------------------------------------------------------- |
| Provider-side text provenance | LLM watermark    | Cheap at generation time and easy to test later ([ar5iv][1])           |
| Platform-side text screening  | DetectGPT        | Useful when no watermark was embedded beforehand ([ar5iv][2])          |
| Model-owner image provenance  | Stable Signature | Every generated image can inherit a recoverable signature ([ar5iv][3]) |

**Information not provided:** none of these papers gives a complete production blueprint for cross-platform provenance standards, legal evidentiary use, large-scale moderation pipelines, or cryptographic authentication ecosystems. They provide algorithms and experiments, not end-to-end governance infrastructure. ([ar5iv][1])

## Limitations and Trade-offs

The biggest trade-off is **cooperation versus coverage**. Watermarking methods are powerful when the generator cooperates, but they do nothing for content produced by non-cooperating systems. DetectGPT covers those cases better, but it lacks intentional embedded provenance. ([ar5iv][1])

A second trade-off is **robustness versus quality**. Stronger watermark signals are easier to detect, but if pushed too hard they can degrade generation quality or become easier to notice. The LLM watermark paper explicitly discusses how low-entropy text is especially challenging. Stable Signature also balances message recovery against image fidelity using a perceptual loss. ([ar5iv][1])

A third trade-off is **security versus openness**. The LLM watermark can be operated in public or private mode. Public mode helps third-party verification but gives attackers more information. Stable Signature warns that leaking the extractor can strengthen attacks. ([ar5iv][1])

A fourth trade-off is **statistical evidence versus operational practicality**. DetectGPT is elegant and effective, but it is computationally expensive because it needs many perturbations and repeated scoring. The paper explicitly notes this cost. ([ar5iv][2])

Finally, there is a **modality trade-off**. Text watermarking lives in discrete token choices, while image watermarking in latent diffusion uses learned hidden signatures plus extractors. The high-level goal is similar, but the technical tools are very different because text and images have different generation mechanics. ([ar5iv][1])

* **Stable Signature** fine-tunes the latent decoder of a diffusion model so generated images contain a hidden binary signature recoverable by a watermark extractor and verified statistically. ([ar5iv][3])

### Likely interview questions

#### 1. What is the difference between watermarking and detection?

Watermarking actively inserts a signal during generation. Detection tries to decide whether content is model-generated after the fact, with or without a watermark. The LLM watermark and Stable Signature are watermarking methods; DetectGPT is a detection method. ([ar5iv][1])

#### 2. How does the LLM watermark actually work?

At each token step, it pseudo-randomly partitions the vocabulary into green and red tokens based on the current context, softly boosts green-token logits, samples normally, and later tests whether the generated passage contains too many green tokens to be plausible under ordinary writing. ([ar5iv][1])

#### 3. Why is low-entropy text hard to watermark?

Because when the next token is almost predetermined, the model has little freedom to choose a different token just to satisfy the watermark. Any strong bias could noticeably harm text quality, and such sequences are also hard to distinguish from human writing in principle. ([ar5iv][1])

#### 4. What is DetectGPT’s main idea in one sentence?

Model-generated text tends to sit in regions where small meaning-preserving rewrites reduce the source model’s log probability more than they do for human text. ([ar5iv][2])

#### 5. Why is DetectGPT called zero-shot?

Because it does not train a new detector on labeled human and machine text. It uses the source model and an off-the-shelf perturbation model directly. ([ar5iv][2])

#### 6. What is DetectGPT’s biggest limitation?

It is most effective when you can query log probabilities from the source model or a close surrogate, and it is computationally expensive because it must generate and score many perturbations. ([ar5iv][2])

#### 7. What makes Stable Signature different from ordinary image watermarking?

The watermark is rooted inside the latent diffusion model’s decoder, so all generated images naturally carry the signature without changing the diffusion process or adding a separate post-processing watermark step. ([ar5iv][3])

#### 8. Can Stable Signature do attribution, not just detection?

Yes. The paper explicitly discusses both detection of origin and identification of which distributed model copy generated an image by using distinct signatures. ([ar5iv][3])

#### 9. Which method would you use if the generator did not cooperate?

DetectGPT, because it does not require a watermark to have been embedded beforehand. That said, it still needs scoring access and enough compute. ([ar5iv][2])

#### 10. Which method is strongest for provider-controlled provenance?

For text, the LLM watermark paper is the clearest provider-controlled approach. For latent diffusion images, Stable Signature is the clearest provider-controlled model-rooted approach. That comparison is a synthesis across the papers. ([ar5iv][1])

## Glossary

| Term                         | Beginner-friendly definition                                                                           |
| ---------------------------- | ------------------------------------------------------------------------------------------------------ |
| **Watermark**                | A hidden signal intentionally embedded into generated content so it can later be detected              |
| **Detector**                 | A method that decides whether content is likely machine-generated                                      |
| **Proactive watermarking**   | Watermarking added during generation by a cooperating model provider                                   |
| **Reactive detection**       | Detecting machine-generated content after the fact without requiring a watermark                       |
| **Green list / red list**    | The random split of vocabulary tokens used by the LLM watermark paper to bias generation               |
| **Logit bias**               | A small value added to some token logits to make them more likely to be sampled                        |
| **z-test**                   | A statistical test used to see whether the observed number of green tokens is unusually high           |
| **p-value**                  | A statistical measure of how surprising the observed evidence would be under a null hypothesis         |
| **Probability curvature**    | How the model’s log probability changes around a passage when that passage is slightly perturbed       |
| **Perturbation discrepancy** | DetectGPT’s score measuring how much log probability drops under small rewrites                        |
| **White-box setting**        | A setting where the detector can score text using the source model’s probabilities                     |
| **Latent diffusion model**   | A diffusion model that generates in a compressed latent space and then decodes to pixels               |
| **Latent decoder**           | The part of a latent diffusion model that converts latent representations into images                  |
| **Watermark extractor**      | A network that reads an image and predicts the hidden watermark message                                |
| **Model purification**       | An attack where a watermarked diffusion model is fine-tuned to reduce or remove the embedded watermark |

These definitions are plain-English summaries of terms used across the three papers. ([ar5iv][1])

## Recap

You should now have a clear view of the landscape. **A Watermark for Large Language Models** shows how to intentionally bias generation so AI text becomes statistically traceable later. **DetectGPT** shows how to detect generated text without any watermark by probing the source model’s local probability geometry. **Stable Signature** shows how to root a hidden signature inside a latent diffusion model so every generated image carries provenance information. ([ar5iv][1])

The most important interview-level lesson is that provenance methods differ mainly in **when the signal is created** and **what access the detector assumes**. Watermarks are stronger when the provider cooperates. Zero-shot detectors are more flexible when the provider does not. Model-rooted image signatures add a stronger notion of source attribution. None of these is universally best; each is best under a different deployment and threat model. ([ar5iv][1])
